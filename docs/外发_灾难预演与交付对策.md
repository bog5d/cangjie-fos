# 外发前：灾难预演与交付对策（发版人 / 产品 / 开发共用）

目的：在**同事实际打开 zip 之前**，把「最可能放弃」的点先压掉或写进指引，避免未使用就先差评。

## 1. 红队：哪些事会让人「没试用就不想再用」

| 顺序 | 糟心体验 | 根因 | 预对策 |
|------|----------|------|--------|
| 1 | 解压后不知道点哪 | 没有根目录总说明 | 压缩包**根**必有：`仓颉FOS-使用指引` + `点击开始-仓颉FOS.bat` |
| 2 | 只有 CangJie_FOS 没有 Coach | 发包漏剪 | 打 zip 用 `tools\build_release_zip.ps1 -BuildFrontend`，且脚本**警告**缺 Coach 时**禁止标记为可外发**（发版人自律）或 CI 加 `-ErrorIfNoCoach` |
| 3 | 路径中文 / 过深 / 在网盘同步目录 | 编码与锁文件 | 指引**第一条**写死推荐 `C:\FOS\` |
| 4 | Python 未装、未进 PATH | 环境 | `同事上手指南` 第一步 + 诊断 bat |
| 5 | 能开白屏或旧页 | 未带 `frontend\dist` | 发前必跑 `build_frontend.ps1` 或 `-BuildFrontend` |
| 6 | 一用就 Pitch_Coach 报错 | 双仓不并列 | ready + 页头横幅 + 本指引第二节 |
| 7 | 首次 15 分钟「像死机」 | 依赖下载 | 指引里写**正常**；`一键启动`里已有首启提示 |
| 8 | 设了 API Key 前端 401 | 只配了后端 | 要外发带 Key 的版本，**必须**构建时加 `VITE_...` 或**不外发** Key 模式 |
| 9 | 问负责人只会得到「你再看下」 | 无可截图材料 | 约定反馈：黑色窗口 + `/api/v1/ready` JSON |

## 2. 蓝队：发版人最小自检（建议当成门禁）

- [ ] `C:\FOS\` 类干净路径**模拟解压**，跑 `点击开始` 能出日志并最终可访问 `http://127.0.0.1:8000`
- [ ] 浏览器看**页头无红色阻塞项**；`/api/v1/ready` 中 `ok` 为 true 或与试点约定一致
- [ ] 双仓**并列**、Coach 下存在 `src\`
- [ ] `frontend\dist\index.html` 存在
- [ ] 生成 **`*.sha256`** 一并发出（防换包扯皮）
- [ ] 试点名单上的 Key / 环境变量在 `RELEASE_CHECKLIST.md` 有记录

## 3. 使用阶段（试起来之后）的常见不爽与对策

- **等太久**：在指引里写清**正常耗时区间**与「任务状态」看哪里。  
- **队列满**：说明并发上限设计；同事侧「少并发提交」即可。  
- **网络类**：DashScope/LLM 在指引里**单列**环境变量，避免「都填了还怪产品」。  

## 4. 与「10 分交付」还差的最后一里

- **真 zip**：仓库不能代替「发到你同事手里的那个文件」——必须在发布机**执行** `build_release_zip.ps1` 产出物。  
- **真反馈闭环**：收一轮试点清单（机器 win 版本、是否有杀软、是否公司代理），**下一轮指引**可再收一刀。
