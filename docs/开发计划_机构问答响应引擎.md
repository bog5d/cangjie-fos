# 开发计划：机构问答响应引擎（尽调 + 投后统一架构）

> 状态：**规划中（未动代码）** | 创建：2026-06-03 | 适用版本：v1.8.0 起
> 本文档先冻结**阶段一（尽调 gk 模式 MVP）**的可落地细节；
> 投后场景仅保留架构占位，细节待后续逐条对齐。

---

## 0. 一句话目标

把「尽调响应」和「投后响应」抽象成**同一台引擎**——
机构发来一份问题清单，系统用历史材料和历史问答自动作答、识别缺口、人工协作补全，
最终产出「按问题归档的材料文件夹 + 答复文档」。

---

## 1. 架构总览：共享引擎 + 场景层

```
┌─────────────────────────────────────────────────────┐
│              机构问答响应引擎（共享）                  │
│                                                       │
│  清单解析 → 材料匹配 ┐                                │
│            问答复用 ┘→ 缺口识别 → 人工审核 → 产出     │
└─────────────────────────────────────────────────────┘
        ▲                              ▲
        │                              │
   ┌────┴─────┐                  ┌─────┴──────┐
   │ 尽调场景  │                  │  投后场景   │
   │（阶段一） │                  │ （阶段二）  │
   └──────────┘                  └────────────┘
```

### 1.1 两个正交的维度（重要，别混淆）

很多概念混乱来自把下面两件事当成一件事。它们是**正交**的：

| 维度 | 取值 | 影响什么 | 怎么决定 |
|------|------|---------|---------|
| **材料库布局** | `flat`（平铺）/ `per_institution`（按机构分子文件夹）| 扫描 + 去重逻辑（F1）| **扫描时自动检测** |
| **响应场景** | `dd`（尽调）/ `post_investment`（投后）| 产出侧重 + 独有功能 | 由入口/用户选择 |

- gk001 的"gk 模式"= 材料库布局是 `per_institution`。
- 投后 = 场景是 `post_investment`。
- 二者可任意组合（gk 的投后 = `per_institution` + `post_investment`）。

### 1.2 引擎组件与现有代码的对应

| 引擎组件 | 现有文件 | 阶段一改造 |
|---------|---------|-----------|
| 清单解析 ChecklistParser | `dd_checklist_parser.py` | 不动（已支持 Word/Excel/PDF/文字 + 分块）|
| 材料池 MaterialPool | `dd_index_service.py` | **扩展**：识别 per_institution 布局、记录来源子文件夹、标记加密文件 |
| 材料匹配 Matcher | `dd_match_service.py` | **扩展**：同名去重留最新、加密文件靠文件名参与 |
| 问答复用 AnswerReuser | —（新建）| **新建** `dd_qa_service.py`：从历史补充资料扒「问→答」、出草稿 |
| 缺口识别 GapDetector | `dd_export_service.py` 内的缺失清单 | 沿用 + 区分"缺材料"/"缺答案" |
| 审核会话 ReviewSession | `dd_match_sessions` / `dd_match_items` 表 | **扩展**：加 `folder_layout`、`scenario` 列 |
| 产出 Exporter | `dd_export_service.py` | **扩展**：按问题归档子文件夹 + 答复文档 |

## 2. 阶段一：尽调 gk 模式 MVP（可落地细节）

### F1 — 多机构池合并扫描 + 去重

**目标**：根目录下按机构名分子文件夹，全部合成一个材料池。

**做法**：
1. 扫描时检测**直属子文件夹**：若多数（建议 ≥60%）不是 `备份/temp/archive/.git` 等系统名，
   判定 `folder_layout = per_institution`，否则 `flat`。
2. `per_institution` 时递归进每个机构子文件夹收文件，
   每条索引记录新增 `institution_subfolder`（来源机构名）。
3. **去重**：同一 `filename` 在多个子文件夹出现 → 只保留 `mtime` 最新的一份。
   去重发生在**索引层**（建索引时即去重），下游匹配看到的已是去重后的池子。
4. 打不开的文件标记 `is_encrypted = 1`（见 F3），仍保留在池中靠文件名匹配。

**数据模型**（`dd_asset_index` 新增列，走 migration）：
- `institution_subfolder TEXT DEFAULT ''`
- `is_encrypted INTEGER NOT NULL DEFAULT 0`
- `mtime REAL`（去重比较用，若已有则复用）

**测试**：
- 造 3 个机构子文件夹，含 1 个同名财报副本（不同 mtime）→ 断言索引只留最新 1 份
- 平铺文件夹 → 断言 `folder_layout = flat`，行为与现有一致（回归保护）
- 机构名子文件夹 → 断言 `folder_layout = per_institution`

---

### F2 — 按问题归档导出

**目标**：每条需求一个子文件夹，把匹配文件拷进去。

**导出结构**：
```
导出目录/
  问题01_近三年财务报表/   ← 多文件
  问题02_营业执照/
  问题03_销售模式资料/
  缺失清单.txt
```

**多文件处理**：一条需求匹配到多个文件时，**不自动全拷**，
在审核表里以候选 + 复选框呈现（复用 `candidates_json`），人工打勾后只拷勾选的。

**沿用现有 guard**：单文件 >100MB 跳过、总量 >500MB 终止（`dd_export_service` 已有，不重写）。

**测试**：
- 3 条需求各匹配文件 → 断言生成 3 个 `问题NN_*` 子文件夹，文件落位正确
- 1 条需求匹配 2 文件、只勾 1 个 → 断言只拷勾选的
- 1 条需求无匹配 → 断言该问题不建空文件夹、进缺失清单.txt

---

### F3 — 加密文件（MVP 范围）

**MVP 只做**：
- 检测加密（尝试打开失败 → `is_encrypted=1`）
- **靠文件名匹配**（不读内容）
- 导出时**原样拷出，不解密**
- 审核表里给加密文件一个 🔒 标记，提示"该文件加密，按文件名匹配"

**明确推迟到 v2**：解密读取内容、导出统一重新加密。

**测试**：
- 造一个带密码的 PDF/xlsx（或 mock 打开失败）→ 断言 `is_encrypted=1`、仍按文件名进匹配池
- 导出加密文件 → 断言字节级原样拷贝（不尝试解密）

---

### F4 — 历史问答复用（MVP：单次扒取 + 草稿）

**目标**：把历史给过的文字答案，复用到新机构同类问题上。

**MVP 流程**：
1. 扫描历史补充资料文件（`补充尽调资料*.docx/pdf`），
   AI 提取「问题 → 答案」对，存入 `dd_qa_pairs` 表（按 `folder_root` 归集）。
2. 新清单某条需求 → 在 `dd_qa_pairs` 里语义检索最相近的历史问答。
3. **出草稿**：命中则给草稿，匹配度不高/无命中则结合内容给个初稿，**都标注置信度**。
4. 人工在界面审核/编辑草稿。
5. 确认后，把「问题 + 答复」汇总成一份**答复文档**（见产出）。

**MVP 与 v1.1 的界线**：
- MVP：单次扒取即用，`dd_qa_pairs` 落表（顺手为 v1.1 铺路，但不做跨会话主动推荐）。
- v1.1：持久化知识库 + 跨会话主动复用 + 去重/版本管理。

**数据模型**（新表 `dd_qa_pairs`）：
```
id, tenant_id, folder_root,
source_file,          -- 答案出处文件
question,             -- 提取的问题
answer,               -- 提取的答案
institution_subfolder,-- 原本回答给哪家
confidence,           -- 提取置信度
created_at
```

**测试**：
- mock LLM 从一段文字扒出 2 个问答对 → 断言 `dd_qa_pairs` 落 2 行
- 新需求语义命中历史问答 → 断言草稿带出历史答案 + 置信度
- 无命中 → 断言返回低置信初稿（不硬塞），标记待人工

---

### F5 — 导出文件夹命名确认

**目标**：统一用对方问题名命名文件夹（我方文件名与对方问题名不一致）。

**做法**：
1. 匹配完成后出**命名确认表**：

   | 对方问题名（→文件夹名）| 匹配文件 | 操作 |
   |------|------|------|
   | 近三年财务报表 | 2022/2023/2024财报 | ✓ / 改名 |
   | 营业执照 | 营业执照.pdf | ✓ |

2. 顶部「一键全部确认（用对方问题名）」按钮。
3. 单条可手动改文件夹名；不点确认时用默认规则（对方问题原文截断）。

**测试**：
- 一键确认 → 断言所有子文件夹用对方问题名
- 单条改名 → 断言该文件夹用改后的名、其余不变

---

### 自动路由（材料库布局检测）

- 扫描端点返回 `folder_layout`，前端据此显示提示
  （"检测到按机构分类的材料库，已切换 gk 模式"），并切到对应审核 UI。
- 用户可手动覆盖（万一检测错），但默认自动。

**测试**：扫描端点对两种布局分别返回正确 `folder_layout`。

## 9. 阶段二占位：投后场景（细节待对齐）

> ⚠️ 以下仅为架构占位，**功能细节尚未对齐，待后续逐条拷问**。

投后 = 共享引擎 + `scenario=post_investment` + 三个独有功能：

| 投后独有 | 待对齐的问题 |
|---------|------------|
| 定期/周期自动生成 | 多久一次？谁触发？生成什么？ |
| "投后常问问题"预测模板 | 模板从哪来？固定清单还是从历史投后清单学？ |
| 当前经营数据/进度快照 | 数据从哪取？手填还是接现有 pipeline 数据？ |

**对阶段一的约束**：引擎接口需保证 `scenario` 是一等参数，
产出层（Exporter）、问答层（AnswerReuser）不得把 `dd` 写死，
否则阶段二要返工。阶段一代码评审时专门检查这一点。

## 10. 不做什么（明确边界）

- ❌ 加密文件解密/重新加密（→ v2）
- ❌ 问答知识库跨会话主动推荐（→ v1.1）
- ❌ 投后任何具体功能（→ 阶段二，待对齐）
- ❌ 不改动 zt 平铺模式现有行为（回归保护）
